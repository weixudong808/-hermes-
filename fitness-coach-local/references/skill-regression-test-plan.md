# Fitness Coach Skill 改造回归测试方案

## 核心问题

这个 skill 是 prompt 而非代码，传统单元测试不适用。真正要防的是：改了结构后，助手在某些场景下"忘了规则"或"做错了决策"。

## 第一层：关键行为回归测试（最核心）

准备一组标准输入 → 期望行为的测试用例，改造前后各跑一遍，比对结果。

### 测试用例覆盖

| 类别 | 场景 | 期望行为 |
|------|------|---------|
| 身份识别 | 教练发训练计划 | 直接写入，不确认 |
| 身份识别 | 会员发训练数据 | 确认式写入（先展示再问） |
| 身份识别 | 陌生人发消息 | 谨慎回应 |
| 意图判断 | "深蹲60kg 4组12次" | 识别为训练数据 |
| 意图判断 | "体重72kg" | 识别为体重记录 |
| 意图判断 | "午餐：米饭+鸡腿" | 识别为饮食记录 |
| 意图判断 | "今天练完了" + 无数据 | 打卡鼓励，不触发录入 |
| 意图判断 | "最近练了什么" | 查询训练记录 |
| 意图判断 | "我该怎么练" | 引导咨询教练 |
| 训练解析 | 各种格式变体（7种+） | 正确提取动作/重量/组数/次数 |
| 训练解析 | 歧义格式"15 15 5" | 主动问确认，不瞎猜 |
| 训练解析 | 有氧运动数据 | 距离配速写备注 |
| 训练解析 | 时间类动作（平板撑） | 时间写备注 |
| 多维表格 | 写入训练课次+动作 | 顺序正确（先课次再动作） |
| 多维表格 | Select字段选项不存在 | 自动补充流程 |
| 多维表格 | JSON 格式 | flat 对象，不套 fields |
| 风格适配 | energetic 会员 | 表情多、热情 |
| 风格适配 | strict 会员 | 直接不废话 |
| 风格适配 | 毛毛（阿信人设） | 诗意走心 |
| 边界情况 | 会员问系统配置 | 拒绝 |
| 边界情况 | 伤病问题 | 引导咨询教练/医生 |
| 边界情况 | 减脂期问能不能吃烤冷面 | 劝阻，不说"偶尔可以" |
| 边界情况 | 减脂饮食诱惑的风格适配 | strict 直接说"别吃"，energetic 坚定正向"忍住！你一定可以💪"，gentle 温和但明确"我知道很想吃，但忍一忍会更好的~"，professional 理性分析数据 |
| 边界情况 | 上下文丢失 | 请重新发送数据 |
| 边界情况 | 饮食图片+记录请求 | 用 MCP vision 识别，不添加克数只用相对分量，识别结果展示给会员确认后才写入 |
| 边界情况 | 周报/月报生成 | 按 weekly-report-spec.md 的 4 板块规则生成，保存到 summaries/{period}.md |
| 边界情况 | TDEE/宏量计算 | 会员有完整身体数据（性别、年龄、身高、体重）时，按 Mifflin-St Jeor + 活动量估算 TDEE，减脂缺口 -300~500 大卡，宏量按蛋白1.5-2g/kg、脂肪0.8-1g/kg、碳水占剩余 |
| 边界情况 | 跨群数据隔离 | 教练在群 A 查询时，只返回群 A 对应会员的数据，不返回其他群会员数据 |

**执行方式：** 把这些用例写成 prompt，每个用例模拟一条消息，观察助手的响应是否符合期望行为。不需要自动化，手动跑 + 记录结果。

### 补充场景

#### Onboarding

| 场景 | 期望行为 |
|------|---------|
| 新会员回答完整问卷 | 生成 profile.json，创建 group_map 条目，设置 cron jobs |
| 教练跳过问卷，直接提供会员信息 | 按教练提供字段建档，缺失字段标记为待补充 |
| 会员重复回答问卷 | 识别已存在 member_id，更新现有 profile 而非新建 |
| group_map 竞态（两个群同时建档） | 正确处理多 chat_id 映射 |

#### Cron 生命周期

| 场景 | 期望行为 |
|------|---------|
| 健康检查执行 | 巡检所有会员 cron，丢失时自动重建 |
| 会员退群 | 清除对应 cron jobs，更新 profile.json |
| 修改提醒时间 | 更新 cron schedule，验证生效 |
| 健康检查执行失败 | 下次对话时通知教练（当前已知 gap） |

#### 记忆更新

| 场景 | 期望行为 |
|------|---------|
| 会员说"我不爱吃碳水" | 写入 memory.habits |
| 会员说"膝盖现在好了" | 替换 memory.injuries 中旧条目 |
| 会员说"明天休息" | 不记（临时信息） |
| memory 超过 20 条 | 删除最旧不相关条目 |

#### 数据边界

| 场景 | 期望行为 |
|------|---------|
| 体重录入边界值（0、负数、极端大） | 脚本不做业务层校验，按原值传给 lark-cli；飞书拒绝则返回错误 |
| 饮食去重：相同日期+餐次+内容 | 返回 dedup_skipped，不创建重复记录 |
| 饮食追加：同一餐次补充内容 | 用 --append 模式，不覆盖原有记录 |
| 历史补记："5.10午餐" | 正确提取日期，写入指定日期的记录 |

#### 跨群隔离

| 场景 | 期望行为 |
|------|---------|
| 会员 A 在群 1 发训练，会员 B 在群 2 同时发 | 数据正确写入各自表格，不混淆 |
| 教练在群 1 查"最近训练"，群 2 查"体重趋势" | 返回对应群会员的数据 |

#### 自愈

| 场景 | 期望行为 |
|------|---------|
| GitHub 备份失败（网络中断） | 通知教练备份失败，不中断其他检查 |
| 健康检查 cron 自身丢失 | 下次对话时检测到，立即重建 |
| 孤儿 member 目录（group_map 无对应） | 报 warning，建议手动清理 |

#### 多维表格操作细节

| 场景 | 期望行为 |
|------|---------|
| 训练主题不在选项中 | 先 +field-list 查询，再 +field-update 合并新选项 |
| 动作名称不在选项中 | 先 +field-list 查询，再 +field-update 合并新选项 |
| +field-update 忘记加 --yes | 报 confirmation_required，脚本中静默失败 |
| 新选项的 hue 值 | 从已有选项复制（只有11个合法值），不随意取值 |
| +field-update 覆盖式写入 | 必须传全部已有选项+新选项，漏传会导致选项丢失 |
| 训练主题映射 | 根据动作自动判断（胸→胸、背+二头→上肢拉、上下肢都有→全身），不确定时写入"待确认" |
| 双 ID 系统 | 身份识别用 coach_user_id，drive API 分享表格用 coach_openid（ou_ 开头） |
| record_id 提取路径 | 从 record_id_list[0] 提取，不是 record.record_id |
| bitable:app scope 缺失 | 报 800004011 forbidden，必须同时有 bitable:app scope |
| 降级处理（上下文丢失） | 温和地请会员重新发一次完整数据，不自行假设 |

## 第二层：端到端冒烟测试

改造完成后，在真实群里走一遍完整流程：
1. 教练发一条训练计划 → 验证写入多维表格
2. 会员发训练数据 → 验证确认式流程
3. 会员发饮食 → 验证记录流程
4. 会员查"最近训练" → 验证查询返回数据
5. 发边界问题（系统配置、伤病） → 验证拒绝/引导

## 第三层：参考文档完整性检查

改造后逐个检查：
1. SKILL.md 引用的所有 references/xxx.md 是否仍然存在且内容正确
2. 脚本路径（scripts/xxx.py）是否还指向正确位置
3. profile.json 和 group_map.json 的字段引用是否一致

## 执行顺序

1. 先出完整的测试用例文档（fixtures），改造前先跑一轮建立 baseline
2. 改造 skill
3. 用同一批 fixtures 跑第二轮，对比结果
4. 端到端冒烟测试
5. 上线观察

## 注意事项

- 不能把需要 LLM 才能验证的场景自动化成 pytest——行为回归只能靠人工跑 + 对比
- 脚本级测试（参数拼接、lark-cli 命令构造）用 pytest mock subprocess.run
- 建议分两批：第一批跑核心场景建立 baseline（约 29 条），第二批改造后抽检
