## 八、设计决策记录

### 消息过滤：训练记录的写入权限

**决策：** 训练记录的写入对教练和会员都开放，但触发方式和处理流程不同。

| 发送者 | 触发条件 | 写入方式 | 说明 |
|--------|---------|---------|------|
| 教练 | 消息包含动作名 + 数值 | 直接写入 | 无需确认，教练数据即为权威数据 |
| 会员 | 记录意图关键词 + 训练数据 | 确认式写入 | 先展示识别结果，会员确认后才写入 |

**设计理由：** 会员独自训练（有氧跑步、游泳等）后主动录入数据是高频刚需。确认式写入机制防止闲聊被误录入，同时让会员对自己的数据有最终确认权。

**⚠️ 会员录入与教练录入的区别：**
- 教练备注栏：会员录入时留空
- 缺失字段：会员录入时不确定的数据问会员确认，不自行补默认值
- 动作名称别名：匹配后告知会员（如"平板撑已对应平板支撑"）

**体重、饮食照片、训练数据均支持会员录入（不需要@机器人），规则不变。**

**角色识别方式：** group_map.json 全局配置 `_config.coach_user_id` 存储教练 user_id，通过 `from.user_id` 判断发送者。会员的 user_id 存储在每群的 `member_feishu_id` 字段。

**隐私红线（绝不回答）：**
- 其他会员的信息（姓名、电话、训练数据、剩余课时）
- 教练的总会员数
- 任何涉及第三方隐私的问题
- 标准拒绝话术："这个我得保密哦~ 有什么问题可以直接问小卫教练，他会告诉你 💪"

**教练专属权限（会员不可操作）：**
- 修改会员称呼（"@机器人 把张三的称呼改成小三"）→ 更新 profile.json + group_map.json
- 修改会员风格（"@机器人 把张三改成严格型"）
- 调整提醒时间（"把张三的午餐提醒改成11:30"）→ 详见 10.6
- 开关周报/月报（"给张三关闭周报"）
- 开关饮食提醒（"给张三关闭早餐提醒"）
- 开关体重提醒（"给张三开启每日体重提醒"）
- 会员退群（"张三退群了"）→ 详见 10.7
- 查看其他会员信息

**会员消息处理规则：**
- @机器人 → 正常回答（知识问答、鼓励互动、训练记录确认式写入）
- 未@机器人 → 按风格简短互动（打卡鼓励、关怀等日常互动优先级）
- 包含训练数据 + 有记录意图 → 确认式写入训练记录
- 包含训练数据 + 无记录意图 → 先鼓励/肯定 → 反问会员是否需要记录
- 只说"练完了"等打卡用语但无训练数据 → 正常互动鼓励，不触发录入
- 包含图片（饮食照片）→ 自动写入饮食记录表（不需要@机器人）

#### 体重记录规则（S14）

**触发条件：** 会员发送的消息包含"体重"关键词 + 包含数字（不需要 @机器人）

|-------------|------|

#### S14 细化：什么记录，什么不记录

| 数据类型 | 发送者 | 是否记录 | 触发方式 | 说明 |
|---------|--------|:---:|---------|------|
| 训练计划（动作/重量/组数/次数） | 教练 | ✅ | 直接发，不需@ | 自动解析写入 |
| 训练记录（有氧/力量/运动数据） | 会员 | ✅ | 不需要@机器人 | 确认式写入，见 data-entry-paths.md Step 0 |
| 体重数据 | 会员（含"体重"+数字） | ✅ | 不需要@机器人 | 写入体重记录表 |
| 饮食照片 | 会员 | ✅ | 不需要@机器人 | 写入饮食记录表 |
| 文字版饮食 | 会员 | ✅ | 不需要@机器人 | 写入饮食记录表 |

**饮食照片识别流程（vision 工具链）：**
1. 会员发图片 → Hermes 缓存到 `/root/.hermes/cache/images/img_xxx.jpg`，消息中附带路径
2. 尝试 `vision_analyze(image_url=缓存路径)` 识别食物内容
3. **⚠️ 如果 `vision_analyze` 失败（常见原因：模型订阅套餐未开放视觉权限，429 错误）** → 降级使用 `mcp_zai_vision_mcp_analyze_image(image_source=缓存路径, prompt="识别食物内容")`
4. 根据识别结果写入饮食记录表，格式："午餐：{食物详情}"

**注意事项：** vision 识别结果仅用于辅助记录，如果图片模糊无法识别，直接问会员"这顿吃的啥呀~"即可，不要编造内容。

### 上下文记忆兜底（S11）

**决策：** 不固定提示"消息已重置"，而是按需查找、真正找不到时才告知。

**处理逻辑（按优先级）：**
1. **从当前上下文找** → 找到了就回答
2. **从多维表格查**（训练数据/体重等结构化数据）→ 查到了就回答
3. **都找不到 → 诚实告知：** "不好意思，我刚刚刷新了一下，之前的对话细节可能记不太清了，能再说一下吗？"

**优点：** 不干扰会员（上下文还在时不提示）、数据类问题走表格兜底减少"忘记"。

**触发读取多维表格的条件（仅限以下场景）：**
| 关键词 | 示例 | 动作 |
|--------|------|------|
| "昨天练了什么"/"上次训练"/"之前练的" | "我昨天练了什么来着" | 用 lark-cli 查多维表格的训练课次表 |
| "上周"/"这周" | "这周我练了几次" | 查多维表格统计 |
| "我刚刚说的"/"刚才" | "我刚才问的那个" | session_search 搜索（跨 session） |

**原则：** 会员问历史训练/饮食数据 → 查多维表格（精确数据），不查对话上下文（不可靠）。

### 饮食照片提醒机制

**提醒频率：** 每天按时提醒（早餐/午餐/晚餐），会员不回也发，直到课程结束或群解散。

**体重提醒：** 按问卷第9项配置，可选每日提醒（早8点）/ 每周提醒（周一早上）/ 不需要（默认）。

**提醒权限：** 所有提醒时间由教练控制，会员不能私自调整。会员反馈给教练 → 教练通知机器人 → 机器人执行调整。

**入群问卷：** 完整内容、触发流程、解析逻辑、profile 建档流程详见 **第七节**。问卷共 11 项，问卷数据写入 profile.json 的 `goal`、`meals`、`reminder_freq`、`report_enabled`、`style`、`weight_reminder`、`notes` 字段。

**报告开关逻辑：** `report_enabled` 同时控制周报和月报（一起开、一起关）。
- `report_enabled: true`（默认）→ 正常纳入周报/月报 cron 任务
- `report_enabled: false` → 周报/月报 cron 脚本跳过该会员，不生成不发送
- 教练可随时通过 @机器人 指令修改，如"给张三开启周报" / "张三不需要月报了"
- 会员不能自行修改，必须通过教练

**Cron Job 实现：**
- 每个会员独立的定时任务（按问卷时间创建）
- 提醒内容："该吃饭啦~ 记得拍照发群里 📸"（@会员）
- 教练调整时：通过 @机器人 指令修改，如 "把张三的午餐提醒改成11:30"

### 跨群隔离（框架级保证）

Hermes 给每个群分配独立 session，消息自带 `chat_id`，skill 通过 `group_map.json` 按 `chat_id` 匹配 bitable。**不会串群。**

### 群聊会话上下文（platform 行为）

**已配置 `group_sessions_per_user: false`**（config.yaml → platforms.feishu.group_sessions_per_user），同一个群里所有用户共享一个 session 上下文。这意味着：

- 教练和会员在同一个群时，教练 @机器人 能看到会员之前发的图片和消息
- 会员说"好的"时，机器人能看到之前的对话（如"需要帮你记录吗？"）
- 确认式写入的多轮对话（包括跨用户）是可行的
- **权衡：** 两个人的对话混在一个 session 里，context 填得更快。但每个群只有教练+一个会员，频率不高，影响可控

**配置位置：** `~/.hermes/config.yaml` → `platforms.feishu.group_sessions_per_user: false`

**⚠️ 注意：** 这个行为是 Hermes 平台层面控制的，不是 SOUL.md/SKILL.md 决定的。

### 会员专属记忆系统（holographic fact_store）

**决策：** 用 holographic fact_store 的 entity 维度管理每个会员的专属记忆，不用子 agent，不用独立 memory 文件。

**方案对比（为什么选 fact_store）：**

| 方案 | 问题 |
|------|------|
| 子 agent（delegate_task） | 无持久状态、每次 spawn 全新对话、同步执行跑完即销毁、60 个并发撞限速 |
| 每个 member 目录下 memory.md | 整文件塞上下文、越写越长超限、无交叉关联能力、无冲突检测 |
| **holographic fact_store（✅ 采用）** | entity 维度精准查询、trust 评分自动淘汰低价值事实、reason() 跨实体推理、contradict() 矛盾检测 |

**写入纪律：**
- entity 必须用会员名
- tags 必须带 `member:{member_id}`
- 查询时永远带 entity 或 member tag 限定，防止串数据
- 教练自己的记忆走 `memory` 工具（自动注入），会员的记忆走 `fact_store`（按需查询），两条线互不干扰

**可靠更新的三层架构：**

| 层 | 机制 | 覆盖率 |
|----|------|--------|
| 第一层 | SKILL.md 硬规则：对话中发现会员新信息 → fact_store add | ~85% |
| 第二层 | `auto_extract: true`：会话结束时自动扫对话提取偏好/决策 | 兜底 |
| 第三层 | 健康检查 cron 对话回顾：每天凌晨扫最近 24h 对话，按 sender 提取会员信息补写 | ~98%+ |

**Hermes 记忆系统三条写入路径（不只是提示词）：**

| 路径 | 机制 | 自动？ |
|------|------|--------|
| 镜像（on_memory_write） | 内置 memory 每次写入，代码级 hook 自动复制到 holographic | ✅ 代码层面 |
| 手动 | 模型主动调 `fact_store(action='add')` | 需模型判断 |
| auto_extract | 会话结束时正则匹配用户消息中的偏好/决策，需 `auto_extract: true` | ✅ 可配置 |

**⚠️ auto_extract 两大限制（中文场景尤其重要）：**

1. **只支持英文正则。** 源码 `plugins/memory/holographic/__init__.py` 的 `_auto_extract_facts()` 只匹配 `I prefer/like/love`、`my favorite`、`I always/never` 等英文模式。中文消息（"我喜欢…"、"我习惯…"、"别给我推…"）不会被匹配。**对于中文健身场景，auto_extract 基本无效。**

2. **不区分 sender。** 扫全量对话，群聊中多人对话时可能把教练或系统消息误提取。即使启用，提取结果写入 fact_store 时仍需手动检查 entity 归属。

**结论：三层架构中第二层（auto_extract）对当前系统价值极低，可靠性主要靠第一层（SKILL.md 硬规则）+ 第三层（健康检查 cron 对话回顾）。**

**Hermes 记忆不只是"提示词驱动"。** 常见误解是 Hermes 的记忆完全靠系统提示词要求模型主动写。实际上有三条写入路径：镜像（on_memory_write 代码级 hook）、手动（模型调用 fact_store）、auto_extract（正则提取）。不要向教练简化为"只是提示词"。

**现有相关基础设施：**
- 5.1 节已支持会员人设覆写（Persona Override），通过 `fact_store probe entity=会员名` 检查
- GitHub 备份已包含 `memory_store.db`（每天 03:00 健康检查 cron 用 `sqlite3 .backup` 安全拷贝）

### 主动智能分析与 MCP Server（规划中，尚未实施）

**背景：** 当前系统是"被动响应"模式——会员记录数据后只回复确认，不提供分析建议。目标是升级为"主动输出"模式，记录完成后根据数据趋势给出个性化建议，提升专业度和会员体验。

**规划功能：**

| 触发场景 | 分析内容 | 数据来源 |
|---------|---------|---------|
| 体重记录后 | 体重趋势（7-14天）+ 结合目标给饮食/训练建议 | 体重记录表 + 训练课次表 + 饮食记录表 |
| 饮食记录后 | 当日营养概况 + 近期饮食模式分析 | 饮食记录表 + profile（TDEE） |
| 周报/月报 | 综合训练频率、体重变化、饮食自律度 | 全表扫描 |
| 个性化提醒（未来） | "本周已练3次，继续加油"等数据化提醒 | 训练课次表 |

**架构方案：MCP Server + 本地缓存**

```
阿里云轻量服务器（部署环境）
├── Hermes Agent
├── lark-cli（写入时用，走飞书 API）
├── MCP Server（新增，常驻进程，提供数据查询工具）
└── 本地缓存 SQLite（新增，多维表格数据的本地副本）

飞书云端（不变）
└── 多维表格（数据唯一真相源）
```

**数据流：**
- **写入**：lark-cli → 飞书多维表格 + 同时更新本地缓存（双写）
- **查询**：直接读本地 SQLite 缓存，毫秒级响应，不走网络
- **保底同步**：凌晨健康检查 cron 做一次全量同步，防止缓存漏更新

**MCP Server 计划提供的工具：**
- `get_weight_trend(days)` — 体重趋势（方向、起止值、变化百分比）
- `get_diet_summary(days)` — 饮食概况（记录餐数、营养模式）
- `get_training_summary(days)` — 训练概况（次数、类型分布）
- `get_full_analysis(days)` — 一次返回全部分析数据

**TDEE 存储决策：** 不在 profile.json 中存储 TDEE 计算值，每次需要时从基础数据（性别/年龄/身高/体重）现算。理由：体重变化后 TDEE 需要重算，存了容易过期导致不一致。

**实施阶段（教练倾向一步到位，不急着先跑最小版本）：**
1. 实现 MCP Server + 本地缓存 SQLite
2. 改造现有写入脚本（record_weight.py、record_diet.py、训练写入流程）支持双写
3. SKILL.md 更新：体重/饮食记录后调用 MCP 生成个性化建议
4. 周报/月报迁移到 MCP 数据源
5. 未来：个性化提醒也基于 MCP 数据

**设计原则：**
- MCP Server 与现有系统完全隔离——挂了只影响分析建议，记录功能不受影响
- 缓存出问题直接删掉重建，数据真相在飞书
- MCP Server 是统一数据查询入口，所有需要查数据的功能（分析/周报/月报/提醒）都复用
